## Problem

Customer scenario: two containers in the same Kubernetes pod share an AF_UNIX
SOCK_STREAM connection (server `net.Listen("unix", path)`, client
`net.Dial("unix", path)`) via an emptyDir. Each container is dumped
separately. CRIU fails with:

```
Error (criu/sk-unix.c:907): External socket is used. Consider using --ext-unix-sk option.
Error (criu/mem.c:386): Pid-reuse detection failed: no parent inventory, ...
Error (criu/mem.c:718): Can't dump page with parasite
```

The mem.c errors are cascades from the socket error — fix the socket issue
and they go away.

Passing `--ext-unix-sk` doesn't help because CRIU explicitly rejects stream
external sockets at `criu/sk-unix.c:912`
(`Can't dump half of stream unix connection`). That check exists because an
external peer could observe EOF — but the customer freezes BOTH containers
before dumping, so EOF is never observed in practice.

The customer app: `/Users/fals/projects/gitlab/clm/tests/app/modules/unixsocket/unixsocket.go`
The failing test: `/Users/fals/projects/gitlab/clm/tests/scenarios/external_unix_socket_test.go`

## Constraints the user set

- No new CRIU parameters. Reuse `--ext-unix-sk` — it stays the explicit
  user consent gate.
- No new image-format fields. All needed fields already exist on
  `UnixSkEntry`.
- Restore should "look up" the peer automatically — no rendezvous broker,
  no `--external unix[ino]:path`-style hints.

## Core idea

The user's constraint + the freeze guarantee unlocks a much simpler design
than the wiki's "IN PLANS" sketch:

**The listener's bound path IS the rendezvous.**

- Server's listener restores normally via existing `bind() + listen()` at
  `sk-unix.c:1957-1971`. It's a live AF_UNIX listener after restore.
- Client's connected fd gets restored by a fresh `socket() + connect(path)`
  — the kernel doesn't care that this is a "new" connection; it's
  indistinguishable from any fresh client. Then replay queued bytes via
  `restore_sk_queue`.
- Server's accepted-fd gets restored by blocking on `accept()` on the
  listener. The client's reconnect (from the other `criu restore`
  invocation) unblocks it. Replay queued bytes.
- Cross-`criu-restore`-invocation coordination is zero: it happens
  implicitly via the filesystem + the kernel's listener queue.

No socketpair, no SCM_RIGHTS, no fdstore, no new subcommand, no new flags.

## Dump-side changes (Changes 1–3)

File: `criu/sk-unix.c`

1. **`:912-917` — lift stream rejection for the external path.**
   Keep the `opts.ext_unix_sk || unix_sk_exception_lookup_id(ino)` gate at
   `:905`. Allow `SOCK_STREAM` and `SOCK_SEQPACKET` through when the peer
   has a bound name (the existing `!peer->name` check at `:919` stays).

2. **`:439-442` — stream queue drain is FREE.**
   `dump_sk_queue()` (`criu/sk-queue.c:163`) already works for stream via
   `recvmsg(MSG_PEEK)`. No code change. Verify by reading.

3. **`:966-984` (`fix_external_unix_sockets`) — fix the stub entry.**
   - `:968`: replace `e.type = SOCK_DGRAM` with `e.type = sk->type`.
   - `:969`: replace `e.state = TCP_LISTEN` with the peer's actual state.
   - `:973`: replace `e.peer = 0` with `e.peer = sk->peer_ino` — this is
     the back-link to the in-scope side. Our in-scope entry already points
     at the stub via `ue->peer = sk->peer_ino` at `:400`; this populates
     the reverse edge.

**Do NOT touch `criu/mem.c`.** The mem errors are downstream cascades;
they disappear once dump succeeds.

## Restore-side changes (Changes 4–8)

File: `criu/sk-unix.c`

4. **`:1944-1948` — don't error on external stream peer when
   `--ext-unix-sk` is set and the peer has a name.** Fall through to the
   new restore paths.

5. **New function `open_unixsk_extern_connector`** (~60 lines).
   Called when `ui->peer` is `USK_EXTERN` and `ui->ue->type` is
   `SOCK_STREAM`/`SEQPACKET` and our entry is the "connector side"
   (has a peer with a name to dial).
   - `socket(AF_UNIX, ui->ue->type, 0)`
   - Apply `SkOptsEntry` via existing helpers.
   - `connect(sk, peer->ue->name, ...)` — factor the connect block at
     `:1403-1444` of `post_open_standalone` into a helper
     `do_connect_to_path(ui, peer)` and reuse here.
   - `restore_sk_queue(sk, ui->ue->id)` to replay this side's buffered
     bytes.
   - Return fd.

6. **New function `open_unixsk_extern_acceptor`** (~80 lines).
   Called when `ui->peer` is `USK_EXTERN` and we hold an accepted-fd
   (peer is a nameless accepted-fd on the other side).
   - Find the paired listener in the restored image (match by the
     listener's inode, reachable via `ui`'s bound-path relationship — the
     kernel records an accepted fd's peer as the listener's bound name).
   - Wait until listener is `FLE_RESTORED` + in listen state (extend
     `peer_is_not_prepared` at `:1400` as needed).
   - `accept(listener_fd, ...)` with a generous timeout (e.g., 60s).
   - Apply SkOpts, `restore_sk_queue(accepted_fd, ui->ue->id)`, return fd.

7. **`:1993` (`open_unix_sk`) — route USK_EXTERN + stream to the new
   functions.** Before the existing USK_PAIR_MASTER/SLAVE fork.

8. **Ordering** — extend `peer_is_not_prepared` (`:1210-1215`) so an
   acceptor waits for its associated listener to reach listen state. For
   the cross-invocation case, the blocking `accept()` is the natural sync
   point; no CRIU-internal ordering needed.

## Image format

No `.proto` changes. All required fields already exist on `UnixSkEntry`:
`peer`, `name`, `type`, `state`, `uflags` (USK_EXTERN stays as-is, no new
flag).

## Known risks to validate at step 1

- **Does a dumped accepted-fd have its peer's listener path recorded?**
  The stub carries the peer's name at `:970-971`. But the peer of an
  accepted-fd is the *client*, which is nameless. Meanwhile, the *client's*
  stored `name` field should contain the listener's bound path (kernel
  records it on connect). Verify with `crit decode` on the client-side
  dump — if the client's entry carries the listener's path as its `name`
  OR through a `peer` link to a listener entry, we're good. If not,
  dump-side needs a small enrichment to carry the listener path onto the
  external stub.

- **Can a fresh post-restore `accept()` be paired with a dumped
  accepted-fd purely through queue replay?** i.e. does any kernel-side
  socket state (credentials, connected-peer struct) need explicit
  restoration? Probably not — queue bytes are user-visible state; kernel
  struct is recreated by the connect/accept. Prove empirically at step 3.

- **Multiple concurrent accepted connections**: each stub gets its own
  acceptor call; they all block on the same listener. Order of accept
  matches order of client reconnects, which may mis-pair. Not a problem
  for the customer's single-connection test; worth calling out for later.
